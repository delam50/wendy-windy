# Wendy Knowledge Architecture

Wendy uses a canonical knowledge system so clinic facts do not compete with older Markdown files or blog/resource content.

## Canonical Sources

- `data/generated/clinic-knowledge.json`
  - Canonical structured clinic facts.
  - Includes providers, locations, hours, pricing rules, services, massage therapy, animal chiropractic, booking rules, and safety guidance.
  - Built from `src/lib/clinicKnowledge.ts` and `src/lib/providers.ts`.

- `data/generated/blog-index.json`
  - Canonical blog/resource-card source.
  - Generated by `npm run sync:blogs`.
  - Contains title, URL, canonical URL, slug, category, tags, summary, excerpt, headings, keywords, and searchable text.

- `data/generated/knowledge-index.json`
  - Canonical vector-ready retrieval index.
  - Generated by `npm run build:knowledge`.
  - Contains structured chunks for provider profiles, clinic hours, pricing rules, service info, blog articles, locations, massage therapy, animal chiropractic, safety guidance, and booking info.

- `data/generated/knowledge-manifest.json`
  - Lists active and inactive knowledge files.
  - Documents canonical, generated, legacy, stale, and retrieval-active sources.

## Retrieval Hierarchy

Wendy should use knowledge in this order:

1. Safety and red-flag logic.
2. Structured clinic facts: providers, locations, hours, pricing, services, booking, massage, animal chiropractic.
3. Explicit blog/resource requests using `blog-index.json`.
4. Strict contextual resource recommendations.
5. General answer generation using retrieved context.

Blog content must not override structured facts for provider routing, pricing, hours, safety, or booking.

## Scripts

- `npm run sync:blogs`
  - Crawls the Windy Ridge sitemap for blog/article URLs.
  - Rebuilds `blog-index.json`.
  - Rebuilds the canonical knowledge index.

- `npm run build:knowledge`
  - Rebuilds `clinic-knowledge.json`, `knowledge-manifest.json`, `knowledge-index.json`, and regenerates `blog-knowledge.md` from `blog-index.json`.

- `npm run sync:sitemap`
  - Refreshes sitemap/Jane knowledge, refreshes blogs, and rebuilds the knowledge index.

- `npm run sync:knowledge`
  - Full refresh of sitemap, blogs, and canonical knowledge index.

## Admin Diagnostics

Authorized managers can ask Wendy:

- `Show active knowledge sources`
- `Show knowledge manifest`
- `Which file is canonical for blogs?`
- `Show retrieval matches for dry needling`
- `Why did Wendy not return a dry needling article?`
- `Show provider knowledge for Dr. Michelle`

Admin diagnostics require `WENDY_ADMIN_CODE`. The admin code must stay server-side and must never be exposed in client code.

## Future Vector Search Plan

`knowledge-index.json` is vector-ready. Each chunk includes:

- `embedding_text`
- `embedding_model`
- `embedding_updated_at`
- `vector_id`

Recommended next step:

1. Create a Supabase `pgvector` table for Wendy knowledge chunks.
2. Generate embeddings from `embedding_text` using an OpenAI embedding model.
3. Store `vector_id`, `embedding_model`, and `embedding_updated_at`.
4. Query semantic matches from Supabase first, then apply Wendy’s keyword/provider/pricing/resource reranking rules.

Until pgvector is added, Wendy keeps the current lightweight local keyword/ranking retrieval.
